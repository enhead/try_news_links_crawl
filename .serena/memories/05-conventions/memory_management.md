# 记忆管理规范

> 本文档定义记忆库的组织原则、编写规范和维护指南

## 🎯 核心原则

### 1. 按需读取优先
**AI 读取机制**：
- 通过 `list_memories` 查看所有记忆名称
- 根据**文件名/路径**判断相关性
- **仅读取** 1-3 个最相关的记忆

**设计要求**：
- 文件名即语义，无需打开就能判断内容
- 单文件独立完整，无需联合阅读
- 小文件优先，50-150 行/文件

### 2. 从整体到局部
**目录层次**：
```
README.md                    # 1层：总索引
├── 01-architecture/         # 2层：架构
│   ├── README.md
│   └── layers.md           # 3层：具体主题
└── 02-domain/              # 2层：领域
    ├── crawler/            # 3层：子领域
    │   ├── README.md
    │   └── layers.md      # 4层：具体实现
    └── config/
```

**阅读路径**：
- 新人：README → overview → 架构层 README → 具体主题
- 实现功能：直达子领域 README → 具体实现文件

### 3. 单一职责
**文件职责**：
- 一个文件只讲一个主题
- 控制在 50-150 行
- 超过 200 行的必须拆分

**目录职责**：
- 一个目录聚合一类功能
- 例如：`crawler/` 只包含爬虫相关

## 📏 编写规范

### 文件命名规则

#### 动词+名词模式
```
✅ pruning.md          # 剪枝（机制）
✅ execution_flow.md   # 执行流程
✅ registry.md         # 注册表
✅ abstract_config.md  # 抽象配置

❌ architecture.md     # 太泛化
❌ design.md           # 不明确
❌ misc.md             # 杂项（禁止）
```

#### 路径即语义
```
02-domain/crawler/layers.md
↓        ↓       ↓
领域层    爬虫    层类型

# AI 通过路径即可判断：
# "这是讲领域层中爬虫子系统的层类型实现"
```

### 文件结构

#### 标准模板
```markdown
# 主标题

> 一句话概述（20字内）

## 核心概念（可选）
关键概念解释

## 详细说明
主要内容

## 代码示例（可选）
代码示例

## 相关链接
- [相关主题1](../path/to/file)
- [相关主题2](../path/to/file)
```

#### 行数控制
- **最少**：30 行（少于此考虑合并）
- **目标**：50-150 行
- **最多**：200 行（超过必须拆分）

### 内容编写原则

#### 独立完整性
```markdown
✅ 好的例子：registry.md

# 注册表机制

> 管理新闻源配置类的注册和实例化

## 核心概念
注册表使用装饰器注册配置类...

## 使用方式
@NewsSourceConfigRegistry.register("id")
class Config:
    ...

## 注意事项
- 必须在应用启动时导入
- ...
```

```markdown
❌ 坏的例子：

# 注册表

注册表负责管理配置类。

（需要阅读其他文件才能理解）
```

#### 避免重复
**单一信息源（Single Source of Truth）**：
- 同一信息只在一个地方详细描述
- 其他地方通过链接引用

```markdown
✅ 好的例子：

# config/registry.md
详细讲解注册表机制（100行）

# crawler/README.md
配置管理通过注册表实现，详见 [注册表机制](../config/registry)
```

#### 层次递进
**README 概览 → 具体文件详解**：

```markdown
# crawler/README.md (50行)
爬虫系统由三种层、节点、剪枝机制组成...
详见：
- [layers.md](layers) - 层类型详解
- [nodes.md](nodes) - 节点实现
...

# crawler/layers.md (80行)
## EnumerableLayer
详细讲解枚举层的实现、用法、示例...

## MappingLayer
...
```

### 目录组织规范

#### 编号前缀
```
01-architecture/    # 架构层（最基础）
02-domain/          # 领域层
03-infrastructure/  # 基础设施层
04-application/     # 应用层
05-conventions/     # 开发规范
06-tasks/           # 任务管理
```

**原则**：
- 编号表示阅读优先级
- 01 最基础，06 最具体

#### 子目录结构
```
02-domain/
├── README.md           # 必须：领域层导航
├── crawler/            # 子领域
│   ├── README.md       # 必须：子领域概览
│   ├── layers.md
│   └── nodes.md
└── config/
    ├── README.md
    └── registry.md
```

**规则**：
- 每个目录必须有 README.md
- README 作为索引和概览
- README 控制在 30-60 行

## 🔄 维护指南

### 新增记忆

#### 1. 选择位置
```
问：新增"健康检查调度机制"记忆？
答：
- 属于健康检查子领域
- 路径：02-domain/health_check/scheduling.md
```

#### 2. 检查是否需要拆分
```
如果现有文件 health_check/service.md 超过 200 行：
→ 拆分为：
  - service.md（接口定义，80行）
  - service_impl.md（实现细节，90行）
  - scheduling.md（调度机制，60行）
```

#### 3. 更新导航
```
修改 health_check/README.md：
+ - **[scheduling](scheduling)** - 调度机制
```

### 更新记忆

#### 增量更新
```
✅ 内容微调：直接修改原文件
✅ 新增章节：如未超行数限制，追加到原文件
❌ 内容翻倍：必须拆分文件
```

#### 重构记忆
```
当某个主题内容大幅增加：
1. 创建子目录
2. 拆分为多个小文件
3. 创建子目录的 README
4. 更新父级 README 的链接
```

### 删除记忆

#### 过时内容
```
1. 确认内容已过时
2. 删除文件
3. 从父级 README 中移除链接
4. 搜索其他文件的引用并移除
```

#### 合并记忆
```
如果两个文件内容高度相关且总行数 < 150行：
1. 合并内容到一个文件
2. 删除另一个文件
3. 更新所有引用链接
```

## 📊 统计指标

### 理想指标
- **总文件数**：30-50 个
- **平均行数**：60-80 行/文件
- **目录层次**：3-4 层
- **README 数量**：6-10 个

### 预警指标
- **单文件超过 200 行** → 必须拆分
- **单目录超过 10 个文件** → 考虑子目录
- **目录层次超过 5 层** → 过度设计
- **文件少于 30 行** → 考虑合并

## 🚀 最佳实践

### DO（推荐）
- ✅ 文件名即语义，见名知意
- ✅ 路径表达层次关系
- ✅ README 作为导航索引
- ✅ 代码示例完整可运行
- ✅ 相关链接帮助跳转
- ✅ 单一信息源，避免重复

### DON'T（避免）
- ❌ 文件名泛化（common、utils、misc）
- ❌ 单文件超过 200 行
- ❌ 内容重复冗余
- ❌ 循环引用
- ❌ 孤立文件（无 README 引用）
- ❌ 过度拆分（文件少于 30 行）
